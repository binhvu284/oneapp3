# OneApp - Architecture & Context for AI Agents / Developers

This document serves as the primary technical entry point for any Developer or AI Agent working on **OneApp**. It provides architectural context, codebase structure, and technical guidelines.

## 1. Project Overview & Vision

OneApp is a modular, grid-based "super app" built for modern productivity. It uses a Plugin/Module architecture where mini-apps (OneNote, OneCrypto, OneApp AI) live inside a shared ecosystem.

**Tech Stack:**

- **Frontend:** React 18, Vite, TypeScript
- **Styling:** Tailwind CSS, `radix-ui` primitives, `shadcn/ui` based components.
- **State & Routing:** `react-router-dom` (v6), `@tanstack/react-query` for API/server caching, React Context for UI state.
- **Backend / BaaS:** Supabase (Auth, PostgreSQL DB, Storage).
- **Core Libraries:** `react-grid-layout` (Dashboard), `hookform` & `zod` (Form validation), `framer-motion` & `animejs` (used for micro-animations/transitions).

## 2. Directory Structure

- `/src/pages/`: Contains the route-level components for each application section.
  - Sub-apps are highly decoupled (e.g., `OneCrypto.tsx`, `OneNote.tsx`, `OneAppAI.tsx`).
- `/src/components/`: Reusable UI elements.
  - `/src/components/ui/`: Contains atomic design elements (buttons, inputs, dialogs) mostly generated by shadcn/ui.
  - Module-specific component folders (e.g., `/dashboard/`, `/ai/`, `/crypto/`, `/onenote/`).
- `/src/hooks/`: Custom React hooks mapping to specific business logic (e.g., `useSchemaSync`, `useExternalConnection`).
- `/src/contexts/`: React Context providers for global states like `AuthSourceContext`, `UISettingsProvider`.
- `/src/lib/`: Shared utilities, data-layer setup, and API clients.

## 3. Key Concepts & Mechanisms

### 3.1. Database & Schema

The project relies on Supabase. A comprehensive snapshot of the database structure is provided in `schema.txt`. Notable tables:

- `oneapp_users` vs `auth.users`: Custom user metadata is stored alongside the native auth table.
- **Modules**: `modules`, `module_config`, `in_use_apps` dictating what apps a user has enabled.
- **AI Engine**: Core functionality driven by `ai_agents`, `ai_conversations`, `ai_messages`, and `ai_agent_memory`.

### 3.2. Data Synchronization Framework

As seen in `src/hooks/useSchemaSync.tsx` and `src/contexts/AuthSourceContext.tsx`, OneApp is meant to be highly dynamic, supporting external database connections so that users/developers can potentially point their OneApp instance to their own Supabase project. Always respect this dynamic config rather than hardcoding Supabase URL/keys.

### 3.3. Aesthetic & UI Principles

OneApp heavily emphasizes **premium, modern aesthetics**.

- Deep use of CSS variables (managed in `index.css` and `ThemeSettings`) allows infinite theme configurations.
- Micro-interactions, rounded corners, glassmorphism, and responsive grids are standard.
- _Rule for UI updates:_ Use Radix UI components via the existing `components/ui/` wrappers. Avoid raw HTML inputs/modals unless absolutely necessary.

## 4. Development Workflow & AI Rules

When developing features for OneApp:

1. **No Localhost Left Behind:** Ensure the dev server is only active during active sessions. Avoid memory leaks.
2. **Component Reusability:** Before building a new dropdown or tab, check `src/components/ui`.
3. **Database Migrations:** If a feature requires new tables/columns, update `schema.txt` conceptually to maintain project context context, and provide the SQL for the user to run on Supabase.
4. **Localization:** The primary language for users is Vietnamese, but the application code, variable names, architecture, and developer communication must be kept strictly in English.
